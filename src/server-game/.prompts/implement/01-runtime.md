# 运行时实现提示词

你现在要为 Go 项目 `server-game` 实现并补强“运行时”。

## 背景

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：启动、主循环、网络、协议、定时器、配置、日志。

本模块覆盖 server-game 的进程生命周期、配置日志、网络入口、协议分发、请求串行化、Tick 调度、关闭流程与运行时诊断。GameServer 的 Win32/IOCP/窗口模型只作为运行语义参考；Go 版以 `main`、goroutine、channel、context、`net/http`、C1/C2 TCP server 和 WebSocket 为运行时主体。ConnectServer/JoinServer/DataServer/ExDB/MapServer 通信归 `28-external-comm.md`，运行时只负责启动和关闭外部通信组件；HTTP/WS 管理入口、维护关闭、运行状态展示和后台命令归 `29-ops.md`；封包加密、CheckSum、CRC、包频率和安全处罚策略归 `27-security.md`，运行时只负责在收发包边界调用安全系统。

本提示词来自 `.prompts/todo/01-runtime.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

## 请先阅读 server-game 文件

- game/server.go

- handle/c1c2.go
- conf/config.go

同时阅读 `.prompts/todo/01-runtime.md`，重点关注 `需修正`、`部分覆盖`、`未覆盖` 状态的条目。

## 请参考 GameServer 文件

- GameMain.cpp
- GameServer.cpp

- TServerInfoDisplayer.cpp

参考 C++ 的业务语义和边界，不要照搬 C++ 的全局对象、线程模型和数据结构。Go 版应优先遵守当前 server-game 的包结构、对象协程/主循环模型和已有协议模型。

## 模块边界

本模块只处理“运行时”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。

本模块涉及的主要子模块分类包括：进程入口与服务生命周期、配置与日志初始化、Game 初始化与组件装载、Game.Start 并发结构、ConnectServer 注册运行时、主循环 channel 串行化、Tick 调度与周期任务、TCP C1/C2 连接生命周期。

## 第一阶段目标

稳定启动、主循环、连接生命周期、定时器和关闭流程，优先修复会影响所有模块的运行时边界。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/01-runtime.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/01-runtime.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。
3. 阅读上面列出的 GameServer 文件，提炼业务语义、状态机、校验点和失败路径。
4. 设计最小 Go 接口和数据流，避免把其他模块的完整规则写进本模块。
5. 先修复 `需修正` 项，再补强 `部分覆盖` 项，最后实现第一阶段必要的 `未覆盖` 项。
6. 接入现有调用点，确保重复调用、断线、失败、回滚或关闭流程安全。
7. 增加单元测试或小范围集成测试，覆盖成功路径和关键失败路径。
8. 运行相关测试，最后总结已完成内容、保留内容和风险。

## 重点检查并修复的问题

- 配置与日志初始化 / GameServer.ini 字段绑定：`DBName/DBUser/DBPassword/DBHost/DBPort` tag 写成 `int`，应改为 `ini`。（当前状态：需修正）
- Game.Start 并发结构 / goroutine 错误上报：建议运行时 goroutine 错误进入统一 error channel，而不是局部退出进程。（当前状态：需修正）
- C1/C2 协议路由与响应编码 / 请求体空长度检查：需要先检查 `len(req.Body)>0`，避免空包 panic。（当前状态：需修正）
- HTTP/WS 运行时入口 / WebSocket Origin 策略：生产环境应限制 Origin 或仅在 debug 开启。（当前状态：需修正）
- Shutdown 与资源释放 / 退出等待：固定 sleep 不可靠，应使用 wait group 或 done channel。（当前状态：需修正）
- 进程入口与服务生命周期 / 全服启动编排：补齐启动失败后的统一回滚策略，避免某个服务失败后残留 goroutine。（当前状态：部分覆盖）
- 进程入口与服务生命周期 / Game 运行时启动：明确 `Start` 是否可重复调用，并增加重复启动保护。（当前状态：部分覆盖）
- 进程入口与服务生命周期 / TCP 服务启动：将监听失败、端口占用、关闭时返回值区分为可预期错误和致命错误。（当前状态：部分覆盖）

- 如果暂时无法写自动化测试，在最终回复中说明原因，并给出可手工验证步骤。
