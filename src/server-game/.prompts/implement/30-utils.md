# 基础工具系统实现提示词

你现在要为 Go 项目 `server-game` 实现并补强“基础工具系统”。

## 背景

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：语言包、随机数、数学工具、路径/配置辅助、通用解析辅助、基础诊断 helper。

本模块覆盖语言包、随机数、数学工具、路径/配置辅助、通用解析辅助和基础诊断 helper。基础工具系统不拥有业务规则，不直接修改玩家、对象、道具、地图、经济或运营状态；它只提供可复用的底层能力，供运行时、对象、地图、掉落、技能、脚本、运营等系统调用。
明确排除：`MuHelper`、`MuBot`、`OfflineLevelling` 归 `31-helper.md`；`BotSystem` 归 `32-service-bots.md`，测试 Bot、模拟连接归 `33-test-bots.md`；运营后台、GM 命令、封禁审计归 `29-ops.md`；Lua/WZ/SMD 的业务脚本生命周期归 `26-script.md`。

本提示词来自 `.prompts/todo/30-utils.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

同时阅读 `.prompts/todo/30-utils.md`，重点关注 `需修正`、`部分覆盖`、`未覆盖` 状态的条目。

参考 C++ 的业务语义和边界，不要照搬 C++ 的全局对象、线程模型和数据结构。Go 版应优先遵守当前 server-game 的包结构、对象协程/主循环模型和已有协议模型。

## 模块边界

本模块只处理“基础工具系统”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。

本模块涉及的主要子模块分类包括：模块边界与总入口、语言包、随机数、数学工具、路径与配置辅助、通用解析辅助、基础诊断 helper、与运行时协作。

## 第一阶段目标

先补强语言包、随机池、数学 helper、路径/配置辅助和基础诊断。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/30-utils.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/30-utils.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。
3. 阅读上面列出的 GameServer 文件，提炼业务语义、状态机、校验点和失败路径。
4. 设计最小 Go 接口和数据流，避免把其他模块的完整规则写进本模块。
5. 先修复 `需修正` 项，再补强 `部分覆盖` 项，最后实现第一阶段必要的 `未覆盖` 项。
6. 接入现有调用点，确保重复调用、断线、失败、回滚或关闭流程安全。
7. 增加单元测试或小范围集成测试，覆盖成功路径和关键失败路径。
8. 运行相关测试，最后总结已完成内容、保留内容和风险。

## 重点检查并修复的问题

- 随机数 / 空池处理：空池应返回 `(nil,false)` 或 error。（当前状态：需修正）
- 随机数 / 零权重处理：总权重为 0 时拒绝抽取。（当前状态：需修正）
- 随机数 / 负权重处理：负权重应在 Put 时拒绝。（当前状态：需修正）
- 与运行时协作 / 初始化失败返回：工具初始化错误应交给运行时决定退出。（当前状态：需修正）
- 模块边界与总入口 / 工具系统初始化顺序：明确语言、随机、配置、解析器的初始化时机和失败策略。（当前状态：部分覆盖）
- 模块边界与总入口 / 无业务状态约束：工具系统不得持有玩家、对象、地图、道具生命周期。（当前状态：部分覆盖）
- 模块边界与总入口 / 跨模块调用规范：只允许业务系统调用工具，工具不得反向 import 业务系统。（当前状态：部分覆盖）
- 模块边界与总入口 / 错误模型：基础工具返回明确错误，避免 os.Exit 分散在工具包内部。（当前状态：部分覆盖）

## 验收标准

- 第一阶段目标中的主流程可以跑通，并且失败路径有明确错误或拒绝行为。
- 本模块新增或修改的状态变化是幂等、可回滚或至少不会破坏已有在线流程。
- 已覆盖项不被重复重写；`部分覆盖` 和 `需修正` 项得到可验证补强。
- 跨系统调用边界清楚，没有把其他模块的大量业务规则搬进本模块。
- 现有核心流程不回退，相关测试通过。

## 测试要求

- 至少覆盖一个成功路径和一个失败路径。
- 覆盖第一阶段涉及的关键状态变化或事务边界。
- 如果本模块涉及并发、断线、重复请求、回滚或定时器，必须覆盖对应边界。
- 如果暂时无法写自动化测试，在最终回复中说明原因，并给出可手工验证步骤。
