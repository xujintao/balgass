# 合成系统实现提示词

你现在要为 Go 项目 `server-game` 实现并补强“合成系统”。

## 背景

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：Chaos Box、Jewel Mix、Socket Mix、Pentagram Mix、翅膀合成、合成事务。

本模块覆盖 `GameServer` 中 ChaosBox 主合成、宝石组合/拆分、翅膀/披风/高级翅膀、事件物品、宠物/坐骑、Harmony、380、Socket、Pentagram、概率费用与合成事务。当前 `server-game` 已有部分协议入口、NPC 类型、配置结构和道具字段，但核心合成流程、材料校验、事务处理与结果生成仍基本缺失。

本提示词来自 `.prompts/todo/12-mix.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

## 请先阅读 server-game 文件

- game/object/player/player.go
- game/object/item.go
- game/item
- game/model/model.go

同时阅读 `.prompts/todo/12-mix.md`，重点关注 `需修正`、`部分覆盖`、`未覆盖` 状态的条目。

## 请参考 GameServer 文件

- ChaosBox.cpp
- JewelMixSystem
- PentagramMixSystem
- protocol.cpp 合成协议

参考 C++ 的业务语义和边界，不要照搬 C++ 的全局对象、线程模型和数据结构。Go 版应优先遵守当前 server-game 的包结构、对象协程/主循环模型和已有协议模型。

## 模块边界

本模块只处理“合成系统”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。

本模块涉及的主要子模块分类包括：合成入口/ChaosBox 状态、ChaosBox 主分发、装备强化 +10~+15、翅膀/披风/高级翅膀、事件/宠物/坐骑合成、药水/果实/特殊箱子、宝石组合/拆分、Harmony 再生系统。

## 第一阶段目标

先实现 Chaos Box 最小事务、材料校验、成功率和失败回滚。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/12-mix.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/12-mix.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。
3. 阅读上面列出的 GameServer 文件，提炼业务语义、状态机、校验点和失败路径。
4. 设计最小 Go 接口和数据流，避免把其他模块的完整规则写进本模块。
5. 先修复 `需修正` 项，再补强 `部分覆盖` 项，最后实现第一阶段必要的 `未覆盖` 项。
6. 接入现有调用点，确保重复调用、断线、失败、回滚或关闭流程安全。
7. 增加单元测试或小范围集成测试，覆盖成功路径和关键失败路径。
8. 运行相关测试，最后总结已完成内容、保留内容和风险。

## 重点检查并修复的问题

- 合成入口/ChaosBox 状态 / CGChaosBoxItemMixButtonClick：解析合成按钮请求，校验玩家状态、ChaosBox 状态、混合类型和材料。（当前状态：部分覆盖）
- 合成入口/ChaosBox 状态 / CGChaosBoxUseEnd：关闭 ChaosBox 时清理合成状态、解锁容器并同步物品。（当前状态：部分覆盖）
- 合成入口/ChaosBox 状态 / CheckEmptySpace_MultiMix：多合成前检查背包可用空间，避免成功后无处放置结果。（当前状态：部分覆盖）
- ChaosBox 主分发 / PlusChaosRate 查询：返回 Crywolf 等系统提供的 Chaos 成功率加成。（当前状态：部分覆盖）
- 装备强化 +10~+15 / Level10-Level15 成功率配置：使用 `Level10` 到 `Level15` 配置驱动强化成功率。（当前状态：部分覆盖）
- 装备强化 +10~+15 / AddLuck 幸运加成：装备幸运属性应按配置增加合成成功率，并受上限限制。（当前状态：部分覆盖）
- 装备强化 +10~+15 / Level15Notice：+15 成功时按配置向玩家或全服广播。（当前状态：部分覆盖）
- 翅膀/披风/高级翅膀 / CheckWingItem：判断一代翅膀材料是否合法，用于翅膀合成和高级翅膀前置。（当前状态：部分覆盖）

## 暂不实现

- 不一次性实现 `.prompts/todo/12-mix.md` 中所有子模块。
- 不实现与本模块边界无关的系统，例如完整活动、完整社交、完整经济或完整运营后台，除非当前模块就是该系统。
- 不照搬 GameServer 的全局变量、裸数组、线程锁模型。
- 不为了通过编译而写空实现掩盖真实缺口。
- 不生成或修改 `.prompts/todo/` 文档。

## 验收标准

- 第一阶段目标中的主流程可以跑通，并且失败路径有明确错误或拒绝行为。
- 本模块新增或修改的状态变化是幂等、可回滚或至少不会破坏已有在线流程。
- 已覆盖项不被重复重写；`部分覆盖` 和 `需修正` 项得到可验证补强。
- 跨系统调用边界清楚，没有把其他模块的大量业务规则搬进本模块。
- 现有核心流程不回退，相关测试通过。

## 测试要求

- 至少覆盖一个成功路径和一个失败路径。
- 覆盖第一阶段涉及的关键状态变化或事务边界。
- 如果本模块涉及并发、断线、重复请求、回滚或定时器，必须覆盖对应边界。
- 如果暂时无法写自动化测试，在最终回复中说明原因，并给出可手工验证步骤。

- 修改了哪些核心路径。
- 哪些第一阶段功能已完成。
- 哪些功能明确保留为后续。
- 跑了哪些测试。
- 是否存在未解决风险或需要用户确认的设计点。
