# 宠物与召唤系统实现提示词

你现在要为 Go 项目 `server-game` 实现并补强“宠物与召唤系统”。

## 背景

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：Helper 宠物、坐骑、DarkHorse、DarkSpirit、Fenrir、Muun、召唤技能、召唤物。

本模块覆盖玩家附属实体和附属效果：Helper 宠物、坐骑、DarkHorse、DarkSpirit、Fenrir、Muun、变身戒指、召唤技能和召唤物。宠物与召唤系统不拥有基础物品、背包、公式、Buff 生命周期、合成和普通怪物 AI；它负责宠物启用状态、宠物经验/等级/耐久、宠物命令、Muun 背包/效果/进化/兑换、召唤物创建和清理，再调用对象、道具、公式、经验、技能、Buff、合成等系统完成底层操作。宠物/坐骑合成归 `12-mix.md`，普通 Buff 生命周期归 `13-buffs.md`，普通怪物 AI 归 `25-monster-ai.md`。

本提示词来自 `.prompts/todo/24-pets-summons.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

## 模块边界

本模块只处理“宠物与召唤系统”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。

本模块涉及的主要子模块分类包括：模块边界与总入口、配置与物品识别、UsePet/坐骑启用、Helper 宠物效果、变身戒指与活动戒指、DarkHorse 黑王马、DarkSpirit 黑王鸟、Fenrir 炎狼兽。

## 第一阶段目标

先实现宠物装备/卸下、耐久、基础属性修正和协议同步。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/24-pets-summons.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/24-pets-summons.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。
3. 阅读上面列出的 GameServer 文件，提炼业务语义、状态机、校验点和失败路径。
4. 设计最小 Go 接口和数据流，避免把其他模块的完整规则写进本模块。
5. 先修复 `需修正` 项，再补强 `部分覆盖` 项，最后实现第一阶段必要的 `未覆盖` 项。
6. 接入现有调用点，确保重复调用、断线、失败、回滚或关闭流程安全。
7. 增加单元测试或小范围集成测试，覆盖成功路径和关键失败路径。
8. 运行相关测试，最后总结已完成内容、保留内容和风险。
