# 测试Bot/压测系统实现提示词

你现在要为 Go 项目 `server-game` 实现并补强“测试Bot/压测系统”。

## 背景

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：模拟连接、模拟登录、模拟选角、模拟在线、断线重连、压测、开发调试命令。

本模块覆盖开发和压测用途的模拟玩家能力，包括 fake connection、模拟登录、模拟选角、模拟在线、断线重连、批量压测、调试命令和压测观测。测试Bot/压测系统不是游戏业务 Bot，不提供 Buffer、Trade、BotShop、HideSeek 等地图服务功能；也不是玩家自己的 MuBot/OfflineLevelling 挂机系统。
明确排除：玩家自动挂机归 `31-helper.md`；服务器地图上的服务型 Bot 归 `32-service-bots.md`；运营后台入口和权限审计归 `29-ops.md`；网络协议真实连接归 `01-runtime.md`。

本提示词来自 `.prompts/todo/33-test-bots.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

## 请先阅读 server-game 文件

同时阅读 `.prompts/todo/33-test-bots.md`，重点关注 `需修正`、`部分覆盖`、`未覆盖` 状态的条目。

## 请参考 GameServer 文件

- 无直接等价；注意不要参考 GameServer BotSystem 作为测试 Bot
- GameServer AddBots 语义仅作命名对照

参考 C++ 的业务语义和边界，不要照搬 C++ 的全局对象、线程模型和数据结构。Go 版应优先遵守当前 server-game 的包结构、对象协程/主循环模型和已有协议模型。

## 模块边界

本模块只处理“测试Bot/压测系统”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。
- 不实现 GameServer `BotSystem` 业务 Bot。
- 不实现玩家 MuBot/OfflineLevelling。

本模块涉及的主要子模块分类包括：模块边界与总入口、Bot 管理、fake 连接、生命周期、状态机、模拟登录、模拟选角、模拟在线。

## 第一阶段目标

先实现 BotManager 并发安全、批量创建、状态观测和生产开关。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/33-test-bots.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/33-test-bots.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。
3. 阅读上面列出的 GameServer 文件，提炼业务语义、状态机、校验点和失败路径。
4. 设计最小 Go 接口和数据流，避免把其他模块的完整规则写进本模块。
5. 先修复 `需修正` 项，再补强 `部分覆盖` 项，最后实现第一阶段必要的 `未覆盖` 项。
6. 接入现有调用点，确保重复调用、断线、失败、回滚或关闭流程安全。
7. 增加单元测试或小范围集成测试，覆盖成功路径和关键失败路径。
8. 运行相关测试，最后总结已完成内容、保留内容和风险。
