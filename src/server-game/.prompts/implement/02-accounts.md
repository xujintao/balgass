你现在要为 Go 项目 `server-game` 实现并补强“账号系统”。

## 背景

你现在位于 Go 项目 `server-game` 的仓库根目录：

/home/pi/balgass/src/server-game

另有 C++ 参考项目 `GameServer`：

/home/pi/balgass-igc/igc/9.5.1.15/source/GameServer

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：登录、登出、登录态、账号认证、安全校验、账号在线唯一性、账号模型、账号管理 API、账号仓库归属。

本模块覆盖账号登录、登出、登录态、账号模型、登录包解析、账号认证、安全校验、账号在线唯一性、账号管理 API、账号仓库归属与账号态清理。角色列表、创建/删除/检查/加载角色、角色保存和进入游戏初始化归角色系统。JoinServer/DataServer/中心在线态通信归 `28-external-comm.md`，账号系统只定义登录业务语义和本地状态约束。后台账号创建/查询/删除、封号/解封入口和操作审计归 `29-ops.md`。

本提示词来自 `.prompts/todo/02-accounts.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

## 模块边界

本模块只处理“账号系统”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。

本模块涉及的主要子模块分类包括：协议入口与登录态、登录请求处理、登录校验与安全、账号状态绑定、账号模型与管理 API。

## 第一阶段目标

完成账号创建、列表、删除、登录认证、登录态和唯一在线的最小闭环。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/02-accounts.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/02-accounts.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。

5. 先修复 `需修正` 项，再补强 `部分覆盖` 项，最后实现第一阶段必要的 `未覆盖` 项。
6. 接入现有调用点，确保重复调用、断线、失败、回滚或关闭流程安全。
7. 增加单元测试或小范围集成测试，覆盖成功路径和关键失败路径。

- 登录请求处理 / HWID 解析：当前只解析不校验，需补机器码限制和审计字段。（当前状态：部分覆盖）
