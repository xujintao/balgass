# 组队系统实现提示词

你现在要为 Go 项目 `server-game` 实现并补强“组队系统”。

## 背景

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：组队邀请、响应、队伍列表、退队/踢出、队长、队友坐标、队伍血条、组队匹配、事件入场授权。

本模块覆盖组队邀请、响应、队伍生命周期、成员列表、队长、退队/踢出、队伍聊天、队友坐标/血条/Buff 同步、组队匹配、事件入场授权，以及经验、掉落、任务、地图、Gens 对组队的联动边界。普通聊天和私聊仍归对象系统入口；组队聊天只记录队伍路由能力。

本提示词来自 `.prompts/todo/15-party.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

## 请先阅读 server-game 文件

- handle/c1c2.go
- game/object/player/player.go
- game/model/model.go
- game/object/object.go

同时阅读 `.prompts/todo/15-party.md`，重点关注 `需修正`、`部分覆盖`、`未覆盖` 状态的条目。

## 请参考 GameServer 文件

- Party.cpp
- PartyClass
- protocol.cpp 组队协议
- user.cpp

参考 C++ 的业务语义和边界，不要照搬 C++ 的全局对象、线程模型和数据结构。Go 版应优先遵守当前 server-game 的包结构、对象协程/主循环模型和已有协议模型。

## 模块边界

本模块只处理“组队系统”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。

本模块涉及的主要子模块分类包括：现有入口、队伍管理器、队伍创建、邀请流程、响应流程、成员维护、列表与通知、队伍聊天。

## 第一阶段目标

先实现组队邀请、响应、成员列表、退队/踢出和队长转移。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/15-party.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/15-party.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。
3. 阅读上面列出的 GameServer 文件，提炼业务语义、状态机、校验点和失败路径。
4. 设计最小 Go 接口和数据流，避免把其他模块的完整规则写进本模块。
5. 先修复 `需修正` 项，再补强 `部分覆盖` 项，最后实现第一阶段必要的 `未覆盖` 项。
6. 接入现有调用点，确保重复调用、断线、失败、回滚或关闭流程安全。
7. 增加单元测试或小范围集成测试，覆盖成功路径和关键失败路径。
8. 运行相关测试，最后总结已完成内容、保留内容和风险。

## 重点检查并修复的问题

- 现有入口 / 组队协议路由：Go 侧已有协议表入口，但缺实际业务处理。（当前状态：部分覆盖）
- 现有入口 / 组队经验配置：配置已存在，需由组队模块提供成员遍历给经验系统。（当前状态：部分覆盖）
- 队伍聊天 / 队伍聊天颜色：配置已存在，需在协议下发或消息构造中使用。（当前状态：部分覆盖）
- 现有入口 / 队友坐标开关：实现客户端请求队友坐标同步的开关和定时下发。（当前状态：未覆盖）
- 现有入口 / 对象队伍字段：恢复或重设 Go 侧对象队伍状态字段。（当前状态：未覆盖）
- 队伍管理器 / PartyClass 初始化：新建队伍管理器，负责队伍数组、ID 分配和并发安全。（当前状态：未覆盖）
- 队伍管理器 / 队伍槽清理：清空队伍成员、等级、事件授权、队长等状态。（当前状态：未覆盖）
- 队伍管理器 / 是否有效队伍：判断 party number 是否有效且成员数大于 0。（当前状态：未覆盖）

## 暂不实现

- 不一次性实现 `.prompts/todo/15-party.md` 中所有子模块。
- 不实现与本模块边界无关的系统，例如完整活动、完整社交、完整经济或完整运营后台，除非当前模块就是该系统。
- 不照搬 GameServer 的全局变量、裸数组、线程锁模型。
- 不为了通过编译而写空实现掩盖真实缺口。
- 不生成或修改 `.prompts/todo/` 文档。

## 验收标准

- 第一阶段目标中的主流程可以跑通，并且失败路径有明确错误或拒绝行为。
- 本模块新增或修改的状态变化是幂等、可回滚或至少不会破坏已有在线流程。
- 已覆盖项不被重复重写；`部分覆盖` 和 `需修正` 项得到可验证补强。
- 跨系统调用边界清楚，没有把其他模块的大量业务规则搬进本模块。
- 现有核心流程不回退，相关测试通过。

## 测试要求

- 至少覆盖一个成功路径和一个失败路径。
- 覆盖第一阶段涉及的关键状态变化或事务边界。
- 如果本模块涉及并发、断线、重复请求、回滚或定时器，必须覆盖对应边界。
- 如果暂时无法写自动化测试，在最终回复中说明原因，并给出可手工验证步骤。
