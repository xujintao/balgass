# 对象系统实现提示词

你现在要为 Go 项目 `server-game` 实现并补强“对象系统”。

## 背景

你现在位于 Go 项目 `server-game` 的仓库根目录：

/home/pi/balgass/src/server-game

另有 C++ 参考项目 `GameServer`：

/home/pi/balgass-igc/igc/9.5.1.15/source/GameServer

`server-game` 参考 C++ `GameServer` 重构而来。本模块在总表中的定位是：对象池、玩家/怪物/NPC、对象移动、对象视野、对象攻击、对象行为、对象道具、对象状态。

本模块覆盖 server-game 的对象运行时承载层，粒度按现有 Go 函数和 GameServer 对象系统语义拆分。对象系统只负责对象生命周期、状态承载、移动/视野/攻击/技能/道具/交互入口；完整地图规则、道具规则、公式、经验、技能定义、任务、合成、掉落表和社交系统分别归入其他正式模块。移动异常、攻击速度、技能距离、安全区违规和处罚状态归 `27-security.md`，对象系统在入口处调用安全检查。GM 移动、召唤、踢人、禁言等运营命令由 `29-ops.md` 发起，对象系统只执行对象级动作。

本提示词来自 `.prompts/todo/04-objects.md`，但不是要求一次实现全部 TODO。你需要把 TODO 转换成一次可闭环、可测试、风险可控的第一阶段开发任务。

## 模块边界

本模块只处理“对象系统”第一阶段所需的核心能力。跨系统能力只能调用对应系统或预留接口，不要顺手实现其他大模块。

本模块涉及的主要子模块分类包括：对象管理器、Object 基础结构、延迟消息与重生状态、Player 运行时、Player 角色承载入口、Monster 运行时、Monster 行为函数、对象移动。

## 第一阶段目标

稳定对象池、玩家/怪物生命周期、移动与视野主链路，避免重构时扩大到完整地图/道具/社交。

第一阶段必须能独立验收。不要试图一次性实现 `.prompts/todo/04-objects.md` 中所有条目。

## 实现要求

- 不修改 `handle` 层协议表，除非第一阶段目标明确需要接入已存在协议且没有其他入口。
- 不跨模块实现完整业务；需要其他模块能力时，优先调用已有接口或定义窄接口。
- 保持当前项目风格，避免引入大型框架或全局 service locator。
- 对状态变化、事务、回滚、并发访问和失败路径给出清晰处理。
- 对尚未支持的复杂功能保留明确扩展点，不要静默吞掉错误。
- 直接修改代码时，必须补充或更新相应测试；如果无法测试，要说明原因。

## 推荐实现顺序

1. 阅读 `.prompts/todo/04-objects.md`，按状态筛出 `需修正`、`部分覆盖` 和第一阶段必须的 `未覆盖` 条目。
2. 阅读上面列出的 server-game 文件，确认当前入口、数据结构、协议结构和已有测试。
